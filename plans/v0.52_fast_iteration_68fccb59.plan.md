# v0.52 -- Fast Iteration

## Problem

`./a run hello.a` takes ~1-2 seconds every time (codegen subprocess + gcc + execute), even when the source hasn't changed. For an AI agent doing write-run-fix loops, this latency adds up fast.

## Changes

### 1. Compilation cache -- [src/cli.a](src/cli.a)

**Cache directory:** `.a_cache/` in the current working directory. Add to [.gitignore](.gitignore).

**Cache key:** SHA-256 hash of the source file content. For programs with `use` imports, the cache must also include imported module content. A pragmatic v1 approach: hash only the main source file. This catches 90% of iteration loops (editing the file you're running). A future version can hash the full dependency tree.

**Cache flow for `cmd_run`:**

```mermaid
flowchart TD
    readSrc["Read source file"] --> hashSrc["hash = sha256(source)"]
    hashSrc --> checkBin{"`.a_cache/{hash}` exists?"}
    checkBin -->|yes| runCached["Execute cached binary"]
    checkBin -->|no| codegen["Codegen + gcc"]
    codegen --> storeBin["Copy binary to `.a_cache/{hash}`"]
    storeBin --> runCached
```

The cached binary lives at `.a_cache/{sha256_hex}`. On cache hit, skip codegen and gcc entirely -- just run the binary. On cache miss, do the normal codegen+gcc, then copy the result into the cache before running.

**Key implementation details in `cmd_run`:**
- Read source with `io.read_file(source_path)`
- Compute `hash.sha256(source)` for the cache key
- Check `fs.exists(cache_bin_path)` for cache hit
- On miss: codegen, gcc, copy binary to cache, then run
- On hit: run the cached binary directly
- `cmd_build` does NOT use the cache (it always produces a fresh output at the user-specified path)

**Cache management:** Add `a cache clean` subcommand that deletes `.a_cache/` entirely. Simple and sufficient.

### 2. `a eval "expr"` -- [src/cli.a](src/cli.a)

Wraps the expression in a minimal program and runs it through the normal codegen+run pipeline:

```
a eval "2 + 3"           ->  wraps as: fn main() { println(2 + 3) }
a eval "str.upper(\"hi\")"  ->  wraps as: fn main() { println(str.upper("hi")) }
```

Implementation: `cmd_eval(expr)` generates a string `fn main() {\n  println({expr})\n}`, writes it to a temp file, calls `cmd_run` on it, cleans up. This reuses the entire compilation pipeline (and the cache).

### 3. Shebang support -- [src/cli.a](src/cli.a)

When `./a run script.a` reads a file starting with `#!`, the first line must be stripped before parsing. The C codegen / parser would choke on `#!`.

Implementation: In `_generate_c()`, after reading the source, check if it starts with `#!` and strip the first line. This lets files like:

```
#!/usr/bin/env a run
fn main() { println("hello") }
```

work both as `./a run script.a` and as `./script.a` (when the file is `chmod +x`).
