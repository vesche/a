# v0.53 -- HTTP Server

## Architecture

```mermaid
flowchart LR
    client[HTTP Client] -->|"TCP connection"| accept["accept()"]
    accept --> parse[Parse HTTP request]
    parse --> req["Build AValue map: method, path, headers, body"]
    req --> call["a_closure_call(handler, 1, req)"]
    call --> resp["Read response map: status, headers, body"]
    resp --> send["Write HTTP response"]
    send --> accept
```

Single-threaded request loop using POSIX sockets. Each connection is handled sequentially (sufficient for dev servers, webhooks, MCP endpoints). The handler closure follows the existing `AClosureFn(env, argc, argv)` convention -- called via `a_closure_call(handler, 1, req_map)`.

## Changes

### 1. C runtime: `a_http_serve` -- [c_runtime/runtime.c](c_runtime/runtime.c)

New function `AValue a_http_serve(AValue port, AValue handler)`:

- `socket(AF_INET, SOCK_STREAM, 0)` + `setsockopt(SO_REUSEADDR)`
- `bind()` to `0.0.0.0:port`, `listen(backlog=64)`
- Infinite loop: `accept()` -> read request -> parse -> call handler -> send response -> close connection
- **Request parsing** (minimal HTTP/1.1):
  - Read until `\r\n\r\n` to get headers
  - Parse first line: `METHOD /path HTTP/1.1` -> extract method + path (including query string)
  - Parse subsequent header lines into a map
  - If `Content-Length` header present, read that many body bytes
- **Request map** passed to handler: `#{ "method": "GET", "path": "/api/users", "headers": #{...}, "body": "..." }`
- **Response map** expected from handler: `#{ "status": 200, "headers": #{...}, "body": "..." }`
  - `status` defaults to 200 if missing
  - `headers` defaults to empty if missing
  - `body` defaults to empty string if missing
  - Auto-adds `Content-Length` and `Connection: close` headers
- On handler error (non-map return, exception), send 500 Internal Server Error
- Returns void (blocks forever); `_void_builtins` entry needed

Also declare in [c_runtime/runtime.h](c_runtime/runtime.h).

### 2. C runtime: `a_http_serve_static` -- [c_runtime/runtime.c](c_runtime/runtime.c)

`AValue a_http_serve_static(AValue port, AValue dir)`:

A convenience that internally creates a file-serving handler. For each request:
- Join `dir` + request `path`, sanitize (reject `..`)
- If path ends with `/`, append `index.html`
- If file exists, read and serve with appropriate Content-Type (based on extension: `.html`, `.css`, `.js`, `.json`, `.txt`, `.png`, `.jpg`, `.svg`)
- If not found, 404 response

This reuses the same socket loop as `a_http_serve` but with a built-in handler instead of a closure.

- `"http.serve"`, `"http.serve_static"` (both block forever, no return value used)

### 4. Rust VM stubs -- [src/builtins.rs](src/builtins.rs), [src/checker.rs](src/checker.rs)

Add `http.serve` and `http.serve_static` to `is_builtin` and `fn_sigs`. The VM implementation can be a stub that prints a message directing users to the native CLI (the VM doesn't need a full socket server; the HTTP server is a native-only feature).

### 5. Example: JSON API -- `examples/api.a` (new)

A working JSON API in ~15 lines demonstrating `http.serve`:

```
fn main() {
  let mut items = []
  http.serve(8080, fn(req) {
    if req["method"] == "GET" && req["path"] == "/items" {
      ret #{ "status": 200, "body": json.stringify(items),
             "headers": #{ "Content-Type": "application/json" } }
    }
    if req["method"] == "POST" && req["path"] == "/items" {
      let item = json.parse(req["body"])
      items = push(items, item)
      ret #{ "status": 201, "body": json.stringify(item),
             "headers": #{ "Content-Type": "application/json" } }
    }
    ret #{ "status": 404, "body": "not found" }
  })
}
```
