# v0.66 -- Native Toolchain Completion

## Part 1: `a fmt` -- Native formatter

The pieces already exist. [std/compiler/parser.a](std/compiler/parser.a) parses source to AST, [std/compiler/emitter.a](std/compiler/emitter.a) renders AST back to formatted source.

**In [src/cli.a](src/cli.a):**
- Add `cmd_fmt(path)` -- read file, `parser.parse(src)`, check for `ParseError`, `emitter.emit(ast)`, write back (or print to stdout with `-` flag)
- Support directory mode: `a fmt src/` formats all `.a` files recursively
- Add dispatch: `if subcmd == "fmt" { ... }`

**Fix in [std/compiler/emitter.a](std/compiler/emitter.a):**
- Add `ExternFn` emission (currently returns `""` for unknown tags). The AST tag exists in `ast.a` and `cgen.a` handles it -- emitter just needs a case for it.

**In [src/cli.a](src/cli.a):**
- Add `cmd_ast(path)` -- read file, `parser.parse(src)`, `json.pretty(ast)` to stdout
- Simplest subcommand -- purely wiring existing modules

## Part 3: `a check` -- Native type checker

Port key checks from [src/checker.rs](src/checker.rs) (~1029 lines) to a new [std/compiler/checker.a](std/compiler/checker.a).

**Scope for v0.66** (practical subset, not full parity):
- **Builtin signature table** -- map of builtin name to `#{"params": [...], "ret": "...", "effects": [...]}`. Start with the ~100 builtins from `checker.rs`.
- **`check(ast) -> [map]`** -- walk AST, return array of diagnostic maps `#{"line": n, "msg": "...", "severity": "error"|"warning"}`
- **Key checks:**
  - Undefined variable references (scope tracking through let/fn params/for bindings)
  - Arity mismatch on known builtin calls
  - Unreachable code after `ret`/`break`
  - Unused variables (warning)
  - Type annotation mismatches where annotations exist
- **Not in scope for v0.66:** full type inference, effect system, exhaustive match checking. Those can come later.

**In [src/cli.a](src/cli.a):**
- Add `cmd_check(path)` -- parse, check, print diagnostics with line numbers and colors

## Part 4: `a repl` -- Interactive loop

Build on the existing `cmd_eval` pattern (wrap expression in `main`, compile, run):

**In [src/cli.a](src/cli.a):**
- Add `cmd_repl()` -- loop: read line from stdin, compile+run, print result
- **State accumulation:** maintain a list of `let` bindings. Each iteration wraps all prior bindings + the new line in a `main()` function, compiles to a temp binary, runs it, captures the output
- **Smart detection:** if input looks like `let x = ...`, add to bindings. If it looks like an expression, wrap in `println(to_str(...))`. If it starts with `fn`, add to accumulated function definitions.
- **Multi-line:** if a line ends with `{`, accumulate until matching `}` before executing
- **Prompt:** `a> ` for normal, `.. ` for continuation
- **Special commands:** `:quit` to exit, `:clear` to reset state, `:bindings` to show current variables

**Flow for each eval:**
```
a> let x = 42
a> x + 1
43
a> let y = x * 2
a> [x, y]
[42, 84]
```

Internally each iteration generates something like:
```a
fn main() {
  let x = 42
  let y = x * 2
  println(to_str([x, y]))
}
```
Then compiles and runs it via the existing `_codegen_subprocess` + `_gcc` path.
