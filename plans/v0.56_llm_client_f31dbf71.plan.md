# v0.56 -- LLM Client

Pure "a" stdlib module. Zero C runtime changes. Composes `http.post` + `json.parse` + `json.stringify` + `env.get`.

## 1. `std/llm.a` -- the module

### Public API

- **`chat(provider, model, messages, options) -> map`** -- unified interface, returns normalized response
- **`models(provider) -> [str]`** -- list common model names per provider (hardcoded convenience)

### Response format (normalized across all providers)

```
#{ "content": "Paris is the capital...",
   "role": "assistant",
   "model": "claude-sonnet-4-20250514",
   "usage": #{ "input_tokens": 14, "output_tokens": 12 },
   "tool_calls": [],
   "stop_reason": "end_turn" }
```

### Provider implementations (internal)

Each provider has a `_build_*_request` and `_parse_*_response` function pair:

**OpenAI** (`"openai"`)
- Endpoint: `https://api.openai.com/v1/chat/completions`
- Auth: `Authorization: Bearer $OPENAI_API_KEY`
- Messages pass through as-is (OpenAI's format is the base format)
- Tool calls: `tools` in request, `tool_calls` in `choices[0].message`
- Response path: `choices[0].message.content`, `usage.prompt_tokens`/`completion_tokens`

**Anthropic** (`"anthropic"`)
- Endpoint: `https://api.anthropic.com/v1/messages`
- Auth: `x-api-key: $ANTHROPIC_API_KEY`, `anthropic-version: 2023-06-01`
- Must extract `system` message from messages array into top-level `system` field
- `max_tokens` is required (default 4096 if not specified)
- Tool calls: `tools` in request, content blocks with `type: "tool_use"` in response
- Response path: `content[0].text`, `usage.input_tokens`/`output_tokens`

**Google AI** (`"google"`)
- Endpoint: `https://generativelanguage.googleapis.com/v1beta/models/{model}:generateContent?key=$GOOGLE_AI_KEY`
- Messages mapped: `role: "user"/"model"`, `parts: [{ text: "..." }]`
- `temperature`/`max_tokens` go in `generationConfig`
- Response path: `candidates[0].content.parts[0].text`, `usageMetadata.promptTokenCount`/`candidatesTokenCount`

### Options map

Recognized keys in the `options` parameter:
- `"temperature"` -- float, 0.0-2.0
- `"max_tokens"` -- int
- `"tools"` -- array of tool definitions
- `"response_format"` -- e.g. `"json"` for JSON mode
- `"max_retries"` -- int (default 3)
- `"retry_delay_ms"` -- int (default 1000)
- `"api_key"` -- override env var lookup
- `"base_url"` -- override endpoint (for proxies, local models)

### Retry logic

Built into `chat()`. On HTTP 429 or 5xx, retry up to `max_retries` times with exponential backoff starting at `retry_delay_ms`. Uses `time.sleep()`.

### Error handling

Returns `Err(message)` on:
- Missing API key
- HTTP error after all retries exhausted
- Invalid provider name
- Malformed response

User wraps in `try { }` or uses `?` operator.

### Key implementation details

- Literal `{` and `}` in strings must use `\{` / `\}` escapes
- Single quotes use `from_code(39)` / `char_code(c) == 39`
- `type_of()` returns `"str"` not `"string"`
- `http.post(url, body, headers)` returns `#\{ "status": int, "body": str \}`
- Comments use `;` not `//`

## 2. Examples

### `examples/chat.a` (~15 lines)
Simple one-shot chat: read a question from args or hardcode, call `llm.chat`, print the response.

### `examples/agent.a` (~40 lines)
Tool-using agent: define a `search_files` tool, send to LLM, process tool calls in a loop, return final answer. Demonstrates the agentic loop pattern.

## 4. Testing

Since LLM calls require API keys and network access, the test strategy is:
- **Compilation test** in `test_memory.sh`: `use std.llm` compiles and runs without calling any LLM
- **`examples/test_llm.a`**: Tests the request-building and response-parsing internals using mock data (call internal `_build_*` / `_parse_*` functions directly with known inputs). No actual API calls.
- **Manual verification**: `examples/chat.a` with a real API key
