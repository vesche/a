# v0.49 -- The Native CLI

The package manager can wait. The LSP can wait. What can't wait is the answer to: **can I actually use this language?**

Right now, running an "a" program natively requires:

```bash
./ac program.a > program.c
gcc program.c c_runtime/runtime.c -o program -I c_runtime -lm -O1 -Wl,-stack_size,0x8000000
./program
```

That's three commands, six flags, and a magic hex number. No language survives that.

## What v0.49 delivers

A single native binary called `a` (written in "a", compiled to C, built with gcc) that does:

```bash
./a run program.a          # compile to C, build, execute, clean up
./a build program.a        # compile to C, build, leave the binary
./a build program.a -o out # compile to C, build as 'out'
./a test tests/            # find test_*.a files, compile, run, report
./a cc program.a           # just emit C to stdout (what ac does now)
```

One command. No flags to remember. No knowledge of where runtime.c lives.

## Architecture

The CLI is a single "a" source file: `src/cli.a`. It:
1. Parses `args()` to determine the subcommand
2. For `run`/`build`: calls the code generator (already available via `use std.compiler.cgen`), writes the C to a temp file, shells out to `gcc` via `exec()`, optionally runs the result
3. For `test`: scans a directory with `fs.ls`, filters for `test_*.a` files, compiles and runs each one
4. For `cc`: just emits C to stdout (backward-compatible with current `ac`)

The CLI knows where the runtime lives because it's compiled alongside it. The gcc flags, stack size, include path -- all baked into the tool.

### Key files

- **[src/cli.a](src/cli.a)** -- the CLI driver (~150-200 lines of "a")
- **[std/compiler/cgen.a](std/compiler/cgen.a)** -- the existing C code generator (used via `use`)
- **[c_runtime/runtime.c](c_runtime/runtime.c)** / **[c_runtime/runtime.h](c_runtime/runtime.h)** -- the existing C runtime (copied/referenced at build time)

### Build chain

```
cargo run -- run std/compiler/cgen.a -- src/cli.a > cli.c    # bootstrap step 1
gcc cli.c c_runtime/runtime.c -o a -I c_runtime -lm -O1 \
    -Wl,-stack_size,0x8000000                                 # bootstrap step 2
./a run hello.a                                               # it works
```

After this, the native `a` binary can compile itself:
```
./a build src/cli.a -o a2    # self-hosting
./a2 run hello.a             # still works
```

## The proof: site_gen.a compiles natively

[examples/site_gen.a](examples/site_gen.a) is the perfect end-to-end test. It uses 6 std modules (path, datetime, hash, csv, template, encoding), filesystem I/O, JSON stringify, string interpolation, closures, maps, and arrays. If `./a run examples/site_gen.a` works and produces a static site in `/tmp`, the language is real.

## What this skips (deliberately)

- **Package manager**: No packages to manage yet. Premature.
- **LSP**: High value but huge scope. Separate version.
- **WASM**: Separate target, separate version.
- **`fmt`/`check`/`ast` subcommands**: Nice to have but not blocking. The Rust CLI still handles these. We can add them later by having the native CLI shell out to the Rust binary for now, or port the formatter to native.

## Implementation plan

### Part 1: The CLI driver

Write [src/cli.a](src/cli.a) that:
- Parses subcommands from `args()`
- Determines the runtime path relative to the binary (or embeds it)
- For `run`: generates C via cgen, writes to a temp file, calls `gcc`, calls the resulting binary, cleans up
- For `build`: same but keeps the binary
- For `cc`: just prints C to stdout
- For `test`: scans directory, compiles each test file, runs, reports pass/fail
- Handles errors gracefully (file not found, compilation failure, etc.)

The runtime path problem: the CLI needs to know where `runtime.c` and `runtime.h` live. Simplest solution: assume they're in `./c_runtime/` relative to cwd, or accept `--runtime-dir` flag. For installed use, we can embed the runtime path at compile time via a string constant.

### Part 2: Bootstrap the CLI

- Use the VM to compile `src/cli.a` to C
- Build the native `a` binary with gcc
- Verify it can compile and run simple programs
- Verify it can compile and run itself (self-hosting)

### Part 3: The proof

- Run `./a run examples/site_gen.a` and verify it produces correct output
- Run `./a run examples/hello.a` through `./a run examples/fibonacci.a`
- Run `./a test` on a directory of test files
- Document the one-line install in README
