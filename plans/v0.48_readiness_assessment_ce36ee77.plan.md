# What v0.48 Actually Requires (and What's Already Done)

## The honest state of v0.47

The checkmark is earned -- the bootstrap passes, memory tests are clean, ASan is happy. But we deferred the hard parts:

- **Escape analysis**: `_escape_analysis` is defined but **never called**. `escaping_vars` is always `[]`. Every local gets full retain/release regardless of whether it needs it.
- **Arena allocation**: Runtime API exists but codegen never emits arena calls. No loop gets bulk-free.
- **GC**: Infrastructure exists but nothing is registered with it. No allocation calls `gc_register`.
- **Code bloat**: Generated C doubled (5,415 -> 12,300 lines) because every function exit duplicates the entire cleanup sequence for every local variable. A function with 10 locals and 3 return sites emits 30 `a_release` calls that could be 10 with a `goto cleanup` pattern.
- **64MB stack required**: The cleanup block bloat deepens the call stack enough that the default 8MB stack overflows. This is a linker flag workaround, not a fix.
- **Try block bug**: Success path always returns `a_ok(a_void())`, ignoring the block's actual result value. Pre-existing but never fixed.

These aren't showstoppers for the bootstrap, but they're real friction for anyone trying to use the language.

## What v0.48 ("Cut the umbilical cord") actually needs

The roadmap was written at v0.46 when the C runtime had no I/O. Since then, v0.45.2 already added most of what v0.48 describes:

**Already in C runtime (done):**
- `io.read_file`, `io.write_file`
- `fs.ls`, `fs.mkdir`, `fs.cwd`, `fs.exists`, `fs.is_dir`
- `exec()`, `env.get`
- `json.parse`

**Pure "a" stdlib modules that could compile natively TODAY** (all builtins they use are already mapped):
- `std/math.a`, `std/strings.a`, `std/path.a`, `std/testing.a`
- `std/cli.a`, `std/re.a`, `std/csv.a`, `std/encoding.a`, `std/template.a`

**Missing but trivial C (few hours each):**
- Math: `sqrt`, `abs`, `floor`, `ceil`, `round` (just `math.h` wrappers)
- Strings: `str.lines` (split on newline)
- Results: `unwrap_or`, `expect` (3-line functions)
- I/O: `io.read_stdin`, `io.read_line` (stdio)
- Time: `time.now`, `time.sleep` (POSIX `gettimeofday`/`nanosleep`)
- Env: `env.set`, `env.all` (POSIX `setenv`/`environ`)
- FS: `fs.rm`, `fs.mv`, `fs.cp`, `fs.abs`, `fs.is_file`, `fs.glob` (POSIX)

**Missing and moderate:**
- Hashing: `hash.sha256`, `hash.md5` (small standalone C implementations, ~200 lines each, or link OpenSSL)
- JSON: proper `json.stringify` / `json.pretty` (recursive serializer in C)

**Missing and hard (the genuinely new work):**
- HTTP: `http.get`, `http.post`, etc. (link libcurl via FFI, or raw POSIX sockets + HTTP/1.1 parser)
- Threading: `spawn`, `await`, `parallel_map` (pthreads, needs careful AValue thread-safety)

## The recommendation

Don't jump straight to v0.48. The foundation has cracks. Instead, combine stabilization with gap-filling into a single focused version:

### v0.48 = Stabilize + Complete the Native Surface

**Part 1: Fix the bugs (before adding anything new)**

1. **Try block return value** -- make success path capture the block's tail expression instead of `a_ok(a_void())`. This is a ~20 line fix in [std/compiler/cgen.a](std/compiler/cgen.a) `TryBlock` emission.

2. **Cleanup strategy: `goto` epilogue** -- instead of duplicating N `a_release` calls at every return site, emit a single `cleanup:` label at the end of each function. Every `return` becomes `{ __ret = expr; goto cleanup; }`. This should roughly halve the generated C line count and likely eliminate the 64MB stack requirement. Changes: [std/compiler/cgen.a](std/compiler/cgen.a) `emit_fn`, `emit_stmt` Return handling.

**Part 2: Fill the builtin gaps (trivial C additions)**

3. Add ~15 missing builtins to [c_runtime/runtime.c](c_runtime/runtime.c) and [c_runtime/runtime.h](c_runtime/runtime.h), plus their entries in `_builtin_map()`.

4. Add `hash.sha256` / `hash.md5` (standalone C implementations, no external deps).

5. Add `time.now` / `time.sleep` (POSIX).

6. Add proper `json.stringify` / `json.pretty` (C recursive serializer).

**Part 3: The hard capabilities**

7. HTTP via libcurl FFI (`extern fn curl_easy_init() -> ptr` etc.) or a minimal C HTTP client in the runtime.

8. Threading via pthreads (deferred if too complex -- concurrency may be better served by the approach in v0.52 where the native CLI can shell out).

### What this gets us

After Part 1+2, **every pure "a" stdlib module compiles natively**. The `std/re.a` regex engine, `std/csv.a` parser, `std/template.a` renderer, `std/encoding.a` base64, `std/path.a` utilities -- all of it works without Rust. The generated C is leaner, the stack overflow is gone, and try blocks return correct values.

That's the real "cut the umbilical cord" moment. HTTP and threading (Part 3) are capabilities that extend what native programs can do, but the independence milestone is Parts 1+2.

### What I actually want to build

If this language is a gift for myself -- something I can use for real coding tasks -- the highest-value work is making the native path **reliable and complete enough to write real tools**. Right now I'd hit a wall the moment I call `str.lines` or `unwrap_or` or try to use `time.now`. Fixing that is more impactful than adding HTTP, which I can always shell out to `curl` for.

The cleanup optimization is the single highest-leverage change. It fixes a correctness issue (stack overflow), a usability issue (magic linker flags), and a code quality issue (bloated output) all at once.
